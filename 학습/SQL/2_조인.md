# 조인 (JOIN)

## 한 줄 요약
**여러 테이블을 연결해서 한 번에 조회**하는 방법. 사원 테이블에 부서 ID만 있고 부서 이름이 별도 테이블이면, 조인으로 합쳐서 봄.

## 예시 데이터

```
employees                    departments
┌────┬───────┬────────┐      ┌─────────┬──────────┐
│ id │ name  │ dept_id │      │ dept_id │ name     │
├────┼───────┼────────┤      ├─────────┼──────────┤
│ 1  │ Kim   │ 1      │      │ 1       │ IT       │
│ 2  │ Lee   │ 2      │      │ 2       │ HR       │
│ 3  │ Park  │ NULL   │      │ 3       │ Sales    │
└────┴───────┴────────┘      └─────────┴──────────┘
```

목표: "사원 이름 + 부서 이름" 한 번에 조회.

## 조인 종류

### 1. INNER JOIN (교집합)
**양쪽에 매칭되는 것만**.

```sql
SELECT e.name AS emp_name, d.name AS dept_name
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id;
```

결과:
```
┌─────────┬───────────┐
│ Kim     │ IT        │
│ Lee     │ HR        │
└─────────┴───────────┘
```

Park은 dept_id가 NULL이라 매칭 안 됨 → 제외.

### 2. LEFT JOIN (왼쪽 기준)
**왼쪽 테이블은 다 나오고, 오른쪽은 매칭되면 채움 (없으면 NULL)**.

```sql
SELECT e.name, d.name
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id;
```

결과:
```
┌─────────┬───────────┐
│ Kim     │ IT        │
│ Lee     │ HR        │
│ Park    │ NULL      │  ← Park은 부서 매칭 안 돼도 살아남음
└─────────┴───────────┘
```

### 3. RIGHT JOIN (오른쪽 기준)
LEFT JOIN의 반대. 거의 안 씀 — LEFT JOIN으로 테이블 순서 바꾸면 됨.

```sql
-- 거의 같은 동작
SELECT * FROM A RIGHT JOIN B ON ...
SELECT * FROM B LEFT JOIN A ON ...
```

### 4. FULL OUTER JOIN
양쪽 다 살림 (매칭 안 되면 NULL). MySQL에선 직접 지원 X (UNION으로 흉내).

### 5. CROSS JOIN (카르테시안 곱)
조건 없이 모든 쌍. 거의 안 씀.

## 직관 — 벤다이어그램

```
A ∩ B  (교집합) → INNER JOIN
A     (왼쪽 전부) → LEFT JOIN
    B (오른쪽 전부) → RIGHT JOIN
A ∪ B  (합집합) → FULL JOIN
```

## ON vs WHERE

```sql
-- 조인 조건은 ON
FROM employees e
INNER JOIN departments d ON e.dept_id = d.dept_id
WHERE e.salary > 5000;
```

- **ON**: 어떻게 합칠지 (조인 조건)
- **WHERE**: 합친 후 필터링

LEFT JOIN에서는 차이가 큼:

```sql
-- ON에 조건 두면 → 매칭 안 돼도 NULL로 남음
LEFT JOIN d ON e.dept_id = d.dept_id AND d.name = 'IT'

-- WHERE에 두면 → 매칭 안 된 행이 NULL로 변환됐다가 다시 걸러짐
LEFT JOIN d ON e.dept_id = d.dept_id
WHERE d.name = 'IT'   -- INNER JOIN과 같아져버림
```

## 다중 조인

3개 이상 테이블도 가능.

```sql
SELECT u.name, p.product_name, o.quantity
FROM users u
INNER JOIN orders o ON u.id = o.user_id
INNER JOIN products p ON o.product_id = p.id;
```

## SELF JOIN — 자기 자신과 조인

같은 테이블을 두 번 사용 (별칭 필수).

### 예: 사원 - 매니저 관계
```
employees
┌────┬──────┬────────────┐
│ id │ name │ manager_id │
├────┼──────┼────────────┤
│ 1  │ Kim  │ NULL       │
│ 2  │ Lee  │ 1          │  ← Kim이 매니저
│ 3  │ Park │ 1          │
└────┴──────┴────────────┘
```

```sql
SELECT e.name AS emp, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
```

```
emp     manager
Kim     NULL
Lee     Kim
Park    Kim
```

## 자주 쓰는 패턴

### 1. 매칭 안 되는 행 찾기
```sql
-- 부서가 없는 사원
SELECT e.*
FROM employees e
LEFT JOIN departments d ON e.dept_id = d.dept_id
WHERE d.dept_id IS NULL;
```

### 2. 다대다 관계 (중간 테이블)
```sql
-- 학생-과목 관계
SELECT s.name, c.title
FROM students s
INNER JOIN enrollments e ON s.id = e.student_id
INNER JOIN courses c ON e.course_id = c.id;
```

### 3. 한 사용자의 주문 합계
```sql
SELECT u.name, SUM(o.amount) AS total
FROM users u
LEFT JOIN orders o ON u.id = o.user_id
GROUP BY u.id, u.name;
```

## 별칭은 필수

테이블명 길거나 다중 조인 시 별칭 안 쓰면 가독성 나쁨.

```sql
-- 좋음
SELECT e.name FROM employees e;

-- 나쁨 (가능하지만 길어짐)
SELECT employees.name FROM employees;
```

## 흔한 실수

### 1. 조인 조건 빠뜨리기 → 카르테시안 곱
```sql
-- ❌ ON 빠짐 → 모든 쌍 (n×m개)
SELECT * FROM A, B;

-- ✅
SELECT * FROM A INNER JOIN B ON A.id = B.a_id;
```

### 2. LEFT JOIN 후 WHERE에 오른쪽 조건
INNER JOIN처럼 동작해버림. 의도가 LEFT면 ON에.

### 3. NULL 비교
```sql
-- ❌
WHERE x = NULL

-- ✅
WHERE x IS NULL
```

## 신호 — 조인이 필요한 문제

- "**A 테이블의 X와 B 테이블의 Y를 같이**"
- "FK(외래키)로 연결된 다른 테이블 정보"
- "~없는 ~을 찾기" → LEFT JOIN + IS NULL
- 부서명 없이 부서 ID만 있는 등의 정규화된 구조

## 실전 팁

- **INNER JOIN이 기본**, 매칭 안 된 것까지 보고 싶으면 LEFT JOIN
- 별칭은 짧게 (e, d, u, o)
- 다중 조인은 한 줄에 하나씩 (가독성)
- 조인 조건과 필터 조건을 명확히 분리 (ON vs WHERE)

## 대표 문제 유형 (프로그래머스 SQL)

- 없어진 기록 찾기 (LEFT JOIN + IS NULL)
- 있었는데요 없었습니다 (시간 비교 + 조인)
- 보호소에서 중성화한 동물 (조인 + 패턴매칭)
