# 코테 공부 가이드 (입문자용)

코딩테스트를 처음 시작하는 사람을 위한 **학습 방향과 사고법** 가이드입니다. 문법 외우기보다 **"문제를 어떻게 바라봐야 하는가"** 에 초점.

> 함께 보는 문서
> - **문법 사전**: `파이썬_문법_레퍼런스.md` — 내장함수/도구
> - **문제별 회고**: `programmers/LV{N}/문제별_회고.md` — 각 문제에서 배운 점

---

## 1. 코테 공부의 3단계

### 1단계 — 문법 익숙해지기 (LV0)
- 파이썬 기본 문법: 변수, 조건문, 반복문, 함수
- 자료형: 리스트, 문자열, 딕셔너리, 셋
- 입출력 형식 익히기 (프로그래머스는 `solution()` 함수만 작성)
- **목표: 문제 보고 코드로 옮기는 게 막힘없이**

### 2단계 — 자주 쓰는 도구 익히기 (LV0~LV1)
- 내장함수 (`sorted`, `sum`, `enumerate`, `zip` ...)
- 컴프리헨션 (`[x for x in arr]`)
- 슬라이싱 (`arr[::-1]`, `arr[1:5]`)
- `collections.Counter`, `collections.deque`
- **목표: 직접 짜기 전에 "이거 내장함수에 있지 않을까?" 떠올리기**

### 3단계 — 알고리즘과 자료구조 (LV2~)
- 정렬, 이진 탐색, BFS/DFS
- 그리디, DP, 완전 탐색
- 자료구조: 스택, 큐, 힙, 해시
- **목표: 문제 유형 파악 → 적절한 알고리즘 선택**

지금은 **1~2단계** 위주. 3단계는 LV1 후반부터 자연스럽게.

## 2. 문제 풀 때 사고 순서

### Step 1 — 문제 읽기
- **요구사항을 한 줄로 요약** 해보기
- 입력/출력 예시로 동작 이해
- 제한사항(`n` 크기) 확인 ⭐

### Step 2 — 시간복잡도 가늠 ⭐⭐⭐
**`n` 크기를 보고 허용 가능한 복잡도를 미리 결정**.

| n 범위 | 허용 시간복잡도 | 풀이 방향 |
|---|---|---|
| n ≤ 100 | O(n³) 가능 | 삼중 for문 OK |
| n ≤ 10,000 | O(n²) 가능 | 이중 for문 OK |
| n ≤ 1,000,000 | O(n) 또는 O(n log n) | 정렬, 단일 루프 |
| n ≤ 100,000,000 | O(√n) 또는 O(log n) | 이진 탐색, 수학적 통찰 |

**예시**:
- `n ≤ 10⁶`인데 이중 for문 짜려고 했다? → **재고**
- `n ≤ 100`인데 어렵게 풀려고 했다? → **그냥 완전 탐색이 답일 수도**

### Step 3 — 풀이 방향 정하기
- 정렬하면 풀리나? (`sorted()`)
- 약수/배수 문제로 환원되나?
- 자료구조가 필요한가? (해시/스택/큐/힙)
- 완전 탐색이 가능한가? (`itertools.combinations` 등)

### Step 4 — 코드 작성
- **내장함수부터 떠올리기**
- 막히면 작은 입력으로 손으로 추적
- 변수명은 의미 있게 (`a`, `b` 보다 `count`, `result`)

### Step 5 — 검증
- 샘플 케이스 통과
- **엣지 케이스**: n=1, 빈 입력, 최대 입력
- 시간복잡도 다시 확인

### Step 6 — 제출 후 회고
- 통과했다면: 더 짧은 풀이는 없을지? 더 빠른 방법은?
- 실패했다면: 왜 틀렸는지 → `문제별_회고.md`에 기록

## 3. "내장함수 쓰지 말라"는 조언, 진실은?

### 반은 맞고 반은 틀린 조언

**맞는 경우**
- 처음 배우는 알고리즘은 **한 번은 직접 구현** (정렬 알고리즘 등)
- 일부 기업 시험(삼성 SW 역량 등)에서 라이브러리 제한

**틀린 경우 (대부분)**
- 일반 코테(프로그래머스/백준/리트코드)는 **내장함수 다 허용**
- 직접 짠 파이썬 코드는 내장함수보다 **수십~수백 배 느림** → 시간 초과의 주범
- 직접 짜면 **버그 가능성** 증가

### 권장 학습 방식
1. **처음 만나는 알고리즘**: 한 번은 직접 짜보기 (이해용)
2. **그 다음부터는 내장함수**: 검증된 도구 사용
3. **실전**: 무조건 내장함수, 슬라이싱, 컴프리헨션부터

## 4. 자주 빠지는 함정 (체크리스트)

### 정수/반올림
- [ ] "정수 부분", "버림", "내림" → **`int()` 또는 `//`** (`round()` 금지)
- [ ] `round(0.5) = 0`, `round(2.5) = 2` (banker's rounding)
- [ ] `f"{x:.0f}"` 도 반올림 + str 반환

### in-place 메서드
- [ ] `.sort`, `.reverse`, `.append`, `.extend`, `.clear` → **None 반환**
- [ ] `x = arr.reverse()` 같이 받지 말 것
- [ ] in-place 메서드 호출 후 객체 자체를 사용 (`return arr`)
- [ ] `-ed` 함수는 새 객체 (`sorted`, `reversed`) ↔ 동사형 메서드는 in-place

### 문자열
- [ ] 문자열은 **불변** — `+=`로 누적하면 O(n²)
- [ ] 누적 필요하면 `list.append()` 후 `"".join(list)`
- [ ] 뒤집기는 `[::-1]` 가장 간결/빠름
- [ ] 문자열은 `.reverse()` 메서드 **없음** (불변이라)

### 반복문
- [ ] `for x in ???:` — `???`는 **iterable**이어야 (정수면 `range()`로 감싸기)
- [ ] `range(n)`은 `n` **미포함** (n까지 필요하면 `range(n+1)`)
- [ ] 이중 for문 보이면 "진짜 두 번 다 돌 필요?" 의심

### 시간복잡도
- [ ] 코드 짜기 전에 **n 크기 → 허용 복잡도** 가늠
- [ ] 이중 for문 = O(n²) — `n ≤ 10⁴` 까지만 안전
- [ ] `a*b = n` 형태 → **약수 문제로 환원** 가능
- [ ] `list.pop(0)`은 O(n) — 앞에서 빼야 하면 **`deque`**
- [ ] `x in list`는 O(n) — 검색 자주면 **`set`**

### 자료구조 선택
- [ ] 검색이 자주 일어남 → `set` 또는 `dict` (O(1))
- [ ] 양끝에서 추가/제거 → `collections.deque`
- [ ] 가장 작은/큰 K개 → `heapq`
- [ ] 빈도수 → `collections.Counter`
- [ ] 정렬된 데이터 검색 → `bisect`

### 기타
- [ ] 함수 호출은 **`()` 필수** — `print(f)` ≠ `print(f())`
- [ ] 모르는 함수는 **반환값이 새 객체인지 / None인지** 확인
- [ ] 2D 배열 만들 때 `[[0]*n]*m` ❌ → `[[0]*n for _ in range(m)]`
- [ ] 중첩 리스트 복사는 `copy.deepcopy()` 또는 컴프리헨션

## 5. 사고 패턴 — 자주 등장하는 환원

문제를 다른 형태로 바꾸면 쉬워지는 경우들.

### 곱셈/나눗셈 → 약수 문제
- `a * b = n` 형태 → "n의 약수 개수" 문제로 환원
- O(n) 또는 O(√n)으로 해결

### 최대/최소/짝짓기 → 정렬
- "가장 큰 값", "두 수의 합 최대", "그리디" 류
- 보통 `sorted()` 한 줄이 핵심

### 중복 제거/유무 검사 → set
- `len(set(arr))` = 고유 원소 개수
- `set(a) & set(b)` = 공통 원소

### 빈도수/가장 많이 나온 것 → Counter
- `Counter(arr).most_common(1)` = 최빈값

### 양끝 처리 → deque
- BFS, 슬라이딩 윈도우
- `list.pop(0)`은 O(n)이라 시간 초과 위험

### 부분집합/조합 → itertools
- `combinations(arr, k)`, `permutations(arr, k)`
- 직접 재귀 짜기 전에 이거부터 떠올리기

## 6. 학습 페이스 추천

### 초보 (LV0)
- **하루 1~2문제**, 시간 제한 두지 말고 천천히
- 풀고 나면 **회고 작성**: 어떤 부분이 막혔나? 새로 배운 함수는?
- 1주일에 한 번 회고 노트 다시 읽기

### 중급 (LV1~LV2)
- **하루 2~3문제**, 30분 안에 풀이 시도
- 못 풀면 **풀이 보고 이해 → 다음날 다시 풀기**
- 비슷한 유형 묶어서 풀기 (정렬 문제, 해시 문제 등)

### 일반 원칙
- **양 < 질**: 한 문제를 깊게 이해하는 게 10문제 대충 푸는 것보다 나음
- **회고가 핵심**: 푼 문제는 1주일 후 다시 봐도 풀 수 있어야
- **남의 풀이 읽기**: "더 짧은 코드"를 보면서 파이썬다운 패턴 익히기

## 8. 막힐 때 대처법

### 30분 고민해도 안 풀리면
1. **문제를 다시 읽기** — 놓친 조건 있는지
2. **작은 입력으로 손으로 추적** — n=1, 2, 3
3. **풀이 검색** — 부끄러워하지 말 것
4. **풀이 이해 후 다음 날 다시 풀기** — 이게 진짜 학습

### 시간 초과
1. n 크기 vs 시간복잡도 다시 확인
2. 이중 for문이면 → 한 번에 줄일 방법 찾기
3. `list` 검색이면 → `set`으로 변경
4. 문자열 누적이면 → `list.append` + `"".join`

### 부분 점수 (예: 66.7점)
- 샘플은 맞고 채점에서 일부 실패 → **엣지 케이스 누락**
- `round()` vs `int()` 같은 미묘한 차이일 가능성
- n=1, n=최댓값, 음수, 0, 빈 입력 등 시도
